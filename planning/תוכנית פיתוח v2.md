# כלי מיגרציה לסיילספורס — v2 (גנרי, ללא-תלות-טמפלייט)

> [!info] מטרת-העל
> v2 עושה את כל מה ש-[[תוכנית פיתוח|v1]] עושה — מיפוי, פיצול-לאובייקטים, זיהוי-כפילויות,
> יצירת קשרים — אבל **ללא שום תלות בטמפלייט מסוים**. כל טבלה שהמשתמש יעלה תעבור את אותו
> תהליך: מיפוי שדות, פיצול לאובייקטים, יצירת קשרים בין אובייקטים שמוגדרים תוך-כדי-עבודה,
> עם UI הרבה יותר אינטואיטיבי ומפורט. **כל מה שב-`config/template_config.py` קשיח-ללקוח
> נמס לתוך הקלט וה-UI.**

המסמך מאחד את כל החלטות-העיצוב מסבב-הבריינסטורם (2026-06-10/11). מקור-אמת לפרטים גולמיים:
`~/.claude/plans/atomic-frolicking-owl.md`.

## עקרונות-יסוד

- **דטרמיניסטי, ללא AI בזמן-ריצה.** הכלי לא "מנחש" באמצעות מודל — האינטליגנציה מגיעה
  ממבנה-הקלט + UI מודרך + מיפוי-אוטומטי לפי שם (fuzzy, לא LLM).
- **Google Sheets בלבד** כמשטח I/O (service account). אין חיבור ישיר לסיילספורס.
- **ליבה קיימת + מעטפת חדשה:** מנועי `modules/` (dedup, פיצול, פלט, נירמול, ולידציה) כבר
  גנריים ובדוקים (115 טסטים) — **נשמרים**. בונים מחדש רק את ה-config הדינמי, ה-UI והתזמור.
- **מסגרת:** Streamlit בגישה **מודרכת-פשוטה** (השראת Fillout) — "כל אחד יוכל לתפעל". CSS
  לפריסה; רכיב-מותאם רק אם מסך ספציפי מחייב.
- **ייבנה ב-Fable 5** → ה-spec מנוסח מונחה-מטרה, לא פרסקריפטיבי.

> [!warning] חובת-נכונות — backfill ב-upsert
> תא ריק ב-upsert **מוחק** דאטה בסיילספורס. חובה למלא חורים מה-DB (backfill) לפני upsert.
> זה קיים ב-v1 ו**חייב לעבור** למנוע הגנרי — אחרת נמחק נתונים בלי לדעת.

## מודל הקלט

### סוג-הטבלה (טוגל בשלב החיבור)
- **טבלת אובייקט יחיד** — כל העמודות שייכות לאובייקט אחד; בוחרים אובייקט פעם אחת, ובמיפוי
  מקצים רק שדות.
- **טבלה מרובת אובייקטים** — עמודות ממקורות שונים; במיפוי מקצים אובייקט+שדה לכל עמודה.

הטוגל קובע **רק** אם מקצים אובייקט פר-עמודה.

### מוסכמת שורות-הכותרת
שלוש שורות (הכללה של מבנה-הבלוקים של v1):
1. **שם האובייקט** (עברית/אנגלית) — מגדיר את הבלוק. במקום `BLOCK_TO_OBJECT` קשיח.
2. **שם-העמודה של הלקוח** — תצוגה בלבד, לא חייב להתאים לסיילספורס.
3. **API** — לרוב ריק מראש; זה בדיוק שלב-המיפוי (ממולא/מאומת מול ה-FieldDefinition).

### מופע (instance)
לשורה שמכילה **יותר מרשומה אחת מאותו אובייקט** (בעל+אישה = שני Contacts): צ'קבוקס פר-אובייקט
**בכותרת הטאב במיפוי** — "מופיע יותר מפעם אחת בשורה" — חושף עמודת **"מופע" (1/2/3)** בטבלת-המיפוי
של האובייקט. מוסתר כברירת-מחדל. ממפה ישירות על מנוע-הבלוקים הקיים (מופע = בלוק).

## המסכים

### 1. חיבור-קלט
- **3 חיבורים** (קלט / מילון-שדות / DB), כל אחד עם **בורר-גיליון + בורר-לשונית + סטטוס**.
  (בורר-הלשונית מחליף את `TEMPLATE_TAB = "טמפלייט טעינה"` הקשיח.)
- חיווי-**עדכניות-DB** ליד חיבור ה-DB (modifiedTime — "עודכן לפני X ימים").
- **שאילתות** (העתקה ל-Inspector): **בורר-השאילתה למעלה**; מתחתיו, בגילוי-מותנה:
  - "מילון שדות (FieldDefinition)" → בחירת-**אובייקטים** (קובעים את ה-WHERE).
  - "ייצוא DB — {אובייקט}" → **רמיזת-לשונית** ("שמור ללשונית: {אובייקט}").
  - לכל שאילתה: תיבת-עריכה (LTR) + העתקה.

### 2. מיפוי
טבלה: לכל עמודת-לקוח — שדה Salesforce (dropdown), סטטוס, ותצוגה-מקדימה.
- **טאב לכל אובייקט** (צד-לצד). טאב של אובייקט שחסר במילון מסומן **"חסר מילון"** (ברמת-הטאב,
  לא בכל שורה).
- **מיפוי-אוטומטי עם רף-ביטחון:** התאמה חד-משמעית → **תקין** (אוטומטי); התאמה עמומה (כמה
  מועמדים) → **"בדוק התאמה"** (ממתין לאישורך). זו האינדיקציה היחידה ל"הכלי מיפה".
- **סטטוסים (3 בטבלה):** תקין · בדוק-התאמה · בקרה · (ו**לא-רלוונטי** באפור — עמודות-תיאור/מפריד).
- **מקור-מיפוי** (תגית-זעירה תחת השדה): אוטומטי / ידני / מהקובץ.
- **תצוגה-מקדימה חיה** ("דוגמה → אחרי"): תאריך `15/03/2024 → 2024-03-15`, מפת-ערכים
  `חברתי → 012xx (חברתי)`, בוליאני `כן → TRUE`.
- **הוסף שדה** — שדה-יעד ללא עמודת-מקור (בעיקר **ערך קבוע** לכל הרשומות, למשל `LeadSource`).
- שמירת-API ידנית נכתבת לטמפלייט; הרגע הזה מציג הודעה **"נשמר לקובץ-המקור — יישאר גם אחרי איפוס"**.

> [!tip] מפת-ערכים (value map)
> תגית **לחיצה** "מפת-ערכים" על עמודה פותחת **חלונית-צפה במקום**: טבלת `ערך-מקור → ערך-יעד
> (נטען) → שם (תצוגה, לא-נטען)` + **ערך ברירת-מחדל** (ריק = יסומן כבעיה). מכסה "תרגום פיקליסט"
> בלי שליפת רשימות-ערכים מסיילספורס. התאמה מדויקת בלבד (טווחים — נדחה).

### 3. מנגנוני-זיהוי (פר-אובייקט)
- **טאב לכל אובייקט** + כפתור **"הוסף אובייקט"** (בעיקר לאובייקט-יעד של Lookup ש**לא** נטען —
  קיים בסיילספורס, רק מזהים מולו). אובייקטים-נטענים מופיעים אוטומטית.
- לכל אובייקט: מנגנונים **מדורגים לפי עדיפות** (גרירה לשינוי סדר); כל מנגנון = **צירוף שדות
  ב-AND**; "+ שדה" מוסיף שדה לצירוף; "הוסף מנגנון".
- **טוגל "זיהוי כפילויות פנימיות" פר-אובייקט — ברירת-מחדל כבוי.** כבוי → כל שורה נטענת בנפרד
  (שום רשומה לא נעלמת). דלוק → איחוד שורות שחולקות מנגנון, **תמיד שקוף** ("X אוחדו ל-Y · לחץ לראות").
- שימוש כפול: אותם מנגנונים משרתים גם dedup וגם **פתרון Lookups** ("זוהה לפי").

### 4. קשרים ו-Lookups
גישה מודרכת (רשימת-חיבורים, לא קנבס-גרירה).
- **Lookups** — עמודה שמצביעה לאובייקט אחר: בוחרים שדה-מקור (אובייקט+עמודה), אובייקט-יעד,
  שדה-היעד למילוי (למשל `AccountId`), ו**"זוהה לפי"** — ברירת-מחדל = מנגנון-הזיהוי של היעד.
  נפתר ל-Id; "לא נמצאו" מסומן.
- **קשרים / junction גנרי** — מאחד `relationship_builder` + `campaign_member_builder` למושג
  אחד: בוחרים שני אובייקטים, אובייקט-הקשר, ועמודת-בקרה אופציונלית (התניה: צור רק כש=TRUE).
  **בדיקת-קיום חלה על כולם** (מגיע "חינם" מהאיחוד).

> [!note] שדות-הקשר
> אובייקט-הקשר הוא **עוד אובייקט במסך-המיפוי, עם טאב משלו** — שם מגדירים/רואים את שדותיו
> (Type + תוספות). **שני ה-Id נגזרים אוטומטית** (מהקישור+זיהוי), לא ממופים כעמודות.
> מסך-הקשרים = טופולוגיה + בקרה בלבד.

### 5. בנייה ופלט
- אובייקטים ב**סדר-טעינה נגזר אוטומטית** (Contact/Campaign קודם; junctions ממתינים עד
  שה-Id של ההורים נטענו). כל כרטיס: מספר-סדר, סטטוס (נבנה/ממתין), ספירות, לשונית-פלט, צ'יפ-בדיקה.
- **התרעת רענון-DB** כאן (החלטות insert/upsert מסתמכות על ה-DB); + אזהרת-סשן אחרי טעינה
  ("טענת — ה-DB לא משקף, רענן").
- **ולידציה — שורה גלויה** מתחת לכרטיס: "נבדק: תאריכים · Id · בוליאני · מפות-ערכים · Lookups".
  בעיה → צ'יפ **"⚠️ N בעיות"** לחיץ לרשימה מדויקת (מה+איפה).

### מסך "טיפול ידני" — ריבוי-התאמות בלבד
> [!abstract] שלושה מצבי-זיהוי
> - **התאמה אחת** → Upsert (אוטומטי).
> - **יש זיהוי, לא נמצא** → Insert (חדש, בטוח, אוטומטי).
> - **ריבוי התאמות** → **טיפול ידני** (החלטה אנושית).

המסך (פר-אובייקט): לכל מקרה — רשומת-המקור + שדות-הזיהוי שלה + **מועמדי-ה-DB** לבחירה (radio),
או "צור חדש (Insert)" או "דלג". שום דבר לא נטען עד שמכריעים; הבחירה מוזנת חזרה לגריד-הפלט.

### "ללא זיהוי" — סקירה קבוצתית (לא Insert עיוור)
מחיווי-לחיץ בבנייה נפתחת סקירה שמציגה **אילו שדות כן קיימים** ברשומות הללא-זיהוי
("35 עם email · 5 שם-בלבד") + שתי פעולות **קבוצתיות**:
- **הוסף מנגנון** → קופץ למסך-הזיהוי; מקדד את כולן בבת-אחת.
- **טען כחדשות** → Insert לכל מי שבאמת אין לו זיהוי.

רשת-ביטחון: אזהרה בולטת על **כמות-חריגה** של ללא-זיהוי (תופס שדה-מפתח שלא מופה).

## מסע ה-Id (paste-back)

יצירת רשומות חדשות מחזירה Id מסיילספורס שלא ידוע מראש; junctions צריכים אותו. הזרימה:
1. גריד-הפלט נכתב עם עמודת **מפתח-פנימי** (תווית יציבה לכל שורה).
2. המשתמש טוען את הלשונית ב-Inspector; Inspector מוסיף ~4 עמודות, כולל **`__Id`**.
3. כפתור **"הכנס Ids שחזרו מהטעינה"** קורא מחדש, מתאים `מפתח-פנימי → __Id`, ומשחרר junctions מ"ממתין".

הכוונה 3-צעדים על הכרטיס ("טען · Inspector מוסיף __Id · הכנס Ids שחזרו מהטעינה").

## פרופיל-טבלה

> [!success] ה-killer feature ל"כל טבלה שאעלה"
> קובץ-JSON מקומי שאורז את **כל הקונפיגורציה** (סוג-קלט / מיפוי / מנגנונים / lookups / junctions
> / מפות-ערכים), בשם "לקוח + משימה" (למשל "טיק-צ'ק — אנשי קשר + קמפיינים + קשרים"). ללקוח דומה:
> טוענים פרופיל וגמרנו. הכלי **רומז** התאמה ("8/10 עמודות תואמות") אך **לא מחיל לבד** — הבחירה
> של המשתמש. כפתור "שמור פרופיל" + הצעה עדינה חד-פעמית.

## עקרונות UI/UX

- **סרגל-עליון דביק** (נשאר בגלילה), רוחב-מלא, קוביות 0-רווח עם **מידע-חי** בכל קובייה
  (מה מחובר/נבדק/דורש-בדיקה). שפת-צבעים: 🟢 הושלם · 🔴 בעיה (כולל התנתקות/תקלה) · 🔵 פס=נוכחי
  (לבן+פס-כחול) · ⬜ אפור=טרם.
- **צד-לצד** היכן שאפשר — מינימום גלילה אנכית.
- **אינדיקציה מינימלית** — 2–3 מצבים, שפה ויזואלית אחת, בלי זואו של אייקונים.
- המרת **Id 15→18** אוטומטית בבנייה + הודעה גלויה עם **"בטל"** (שכבה נגזרת; מקור לא נדרס).
- פירמוט **בוליאני** לפי DataType (Checkbox) — אותה תבנית כמו תאריכים ב-v1.

## מיפוי ל-v1: מה נשמר ומה נמס

> [!quote] התובנה המרכזית
> המנועים ב-`modules/` כבר גנריים ומקבלים את הספציפיקה **כפרמטרים**. v2 = להחליף מאיפה מגיע
> ה-config (מקובץ-קבוע → לסכמה דינמית מהקלט+UI), לא לכתוב מנועים מחדש.

- **נשמר כמעט-כפי-שהוא:** `identity.compute_key`, `dedup_engine.deduplicate`, `splitter.split_object`,
  `output_writer` (גרידים+צבעים+טיפול-ידני), `formatter`, `validator`, `sheets_io`, `field_dictionary`,
  `query_builder`.
- **מתאחד:** `relationship_builder` + `campaign_member_builder` → בונה-junction גנרי אחד.
- **נמס לחלוטין:** `config/template_config.py` → **סכמת-runtime** דינמית הנבנית מהקלט + בחירות-ה-UI
  (אובייקטים, מיקומי-שורות, בלוק→אובייקט, עמודות-בקרה, מנגנונים, junctions, מפות-ערכים, שמות-לשוניות).
- **משתנה:** `main.py` (תזמור + UI) — נבנה מחדש סביב הסכמה הדינמית.
